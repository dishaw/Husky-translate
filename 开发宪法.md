# 开发宪法

本项目开发以“少破坏、可恢复、能验证”为基本原则。所有改动都应尽量贴合现有 Odoo 模块结构和前端组件风格，避免为了解决单点问题引入大范围重构。

## 基本原则

- 先读代码再动手，确认入口、调用链和已有约定。
- 改动保持小而准，只处理当前问题，不顺手重构无关代码。
- 不回滚、不覆盖他人或用户已有改动，除非明确要求。
- 用户正在使用的文件和数据优先保护，尤其是上传源文件、译文附件、图片、OCR 结果。
- 对可能造成数据丢失的 fallback 要谨慎，不能静默生成“看似成功但内容缺失”的结果。

## Odoo 开发约定

- 修改 Python 文件后，需要升级相关模块并重启 Odoo web 容器。
- 修改 XML、JS、SCSS 等前端资产后，也应升级模块并重启，必要时浏览器 `Ctrl+F5` 强刷。
- 当前主模块为 `llm_translate`。
- 当前数据库名为 `odoo-translate`。
- 当前服务入口为 `http://localhost:8070`。

常用命令：

```powershell
docker compose exec web odoo -c /etc/odoo/odoo.conf -d odoo-translate -u llm_translate --stop-after-init
docker compose restart web
docker compose ps
docker compose logs --tail 40 web
```

## 翻译功能注意事项

- DOCX 重建应优先基于原文件原地替换文字，以保留图片、表格、文本框、页眉页脚和复杂布局。
- 如果原文件包含图片，不能退回到新建空白 DOCX 的方式，否则会丢失 `word/media` 内容。
- 图片 OCR 叠加失败时，允许降级为保留原图但不插入覆盖文字，不能因此丢掉图片。
- 大文件处理要兼顾稳定性和预览体验，图片提取默认应使用有限模式，避免一次性把过多 base64 塞入前端。
- 前端批次翻译间隔应可配置，避免请求过快或过慢；修改后要确认正在翻译时下一批能即时使用新值。

## 验证约定

- 能跑自动检查就跑自动检查；不能跑时要说明原因。
- 模块升级日志中应看到 `Module llm_translate loaded` 和 `Modules loaded`。
- 重启后应确认 `odoo_trans_web` 为 healthy。
- 看日志时要区分重启瞬间的旧连接关闭和真正的启动失败。

## 提交前检查

- `git diff` 确认只包含本次任务相关改动。
- 检查是否误改编码、换行或大段无关内容。
- 检查是否新增了敏感信息、临时文件、调试垃圾。
- 对用户可见的功能，至少确认入口按钮、弹窗、保存逻辑和失败提示都完整。
