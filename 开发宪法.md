# 开发宪法

本项目开发以"少破坏、可恢复、能验证"为基本原则。架构为**轻量前端门户 + KodBox 文档管理 + OnlyOffice Document Server**，前端使用纯原生 HTML/CSS/JS，托管于 nginx:alpine。

## 架构概览

| 组件 | 技术 | 端口 | 说明 |
|------|------|------|------|
| 前端门户 | nginx:alpine | 8070 | 纯静态品牌 landing page（husky.html） |
| 文档管理 | Cloudreve (Go) | 8080 | 文件夹树/拖拽上传/权限/分享/检索 |
| Document Server | OnlyOffice (复用) | 8090 | 文档编辑与渲染引擎（插件定制保留） |

## 基本原则

- 先读代码再动手，确认入口、调用链和已有约定。
- 改动保持小而准，只处理当前问题，不顺手重构无关代码。
- 不回滚、不覆盖他人或用户已有改动，除非明确要求。
- 用户正在使用的文件和数据优先保护，尤其是上传源文件、译文附件、图片、OCR 结果。
- 对可能造成数据丢失的 fallback 要谨慎，不能静默生成“看似成功但内容缺失”的结果。

## 前端开发约定

- 前端为纯原生 HTML/CSS/JS，不依赖任何重型框架（Vue/React）。
- 首页必须极致轻量，确保 200ms 内渲染完毕，绝不引入 OnlyOffice api.js。
- 修改 `translator-portal.html` 后，重启 nginx 容器即可生效。
- 主入口为 `http://localhost:8070` → 自动跳转 `translator-portal.html`。

常用命令：

```powershell
docker compose up -d --build
docker compose restart portal
docker compose ps
docker compose logs --tail 40 portal
```

## 翻译功能注意事项

- DOCX 重建应优先基于原文件原地替换文字，以保留图片、表格、文本框、页眉页脚和复杂布局。
- 如果原文件包含图片，不能退回到新建空白 DOCX 的方式，否则会丢失 `word/media` 内容。
- 图片 OCR 叠加失败时，允许降级为保留原图但不插入覆盖文字，不能因此丢掉图片。
- 大文件处理要兼顾稳定性和预览体验，图片提取默认应使用有限模式，避免一次性把过多 base64 塞入前端。
- 前端批次翻译间隔应可配置，避免请求过快或过慢；修改后要确认正在翻译时下一批能即时使用新值。

## 验证约定

- 能跑自动检查就跑自动检查；不能跑时要说明原因。
- 重启后确认 `husky_portal` 为 healthy。
- 访问 `http://127.0.0.1:8070` 看到首页即表示服务正常。
- 确认 OnlyOffice 在 `http://127.0.0.1:8090` 可访问。
- 看日志时要区分重启瞬间的旧连接关闭和真正的启动失败。

## 提交前检查

- `git diff` 确认只包含本次任务相关改动。
- 检查是否误改编码、换行或大段无关内容。
- 检查是否新增了敏感信息、临时文件、调试垃圾。
- 对用户可见的功能，至少确认入口按钮、弹窗、保存逻辑和失败提示都完整。
